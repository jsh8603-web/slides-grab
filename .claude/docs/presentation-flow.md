## 단계 기록 파일 (progress.md) — 컨텍스트 압축 대비

프레젠테이션 워크플로우 중 **각 Step 완료 시** `slides/프레젠테이션명/progress.md`에 기록한다.
컨텍스트 압축·세션 이어받기 시 이 파일이 현재 상태의 Single Source of Truth.

## 이벤트 발생 → 체크박스 TASK 관리

**각 PHASE에서 규정하는 TASK 이벤트 발생 시, 즉시 체크박스를 추가한다.**
"나중에 기록" 금지 — 이벤트 발생 시점에 `- [ ]` 즉시 추가.
완료 시 `[x]` 체크. `[ ]` 미완료 항목은 세션 복원·프로덕션 완료 시 반드시 재확인.

### 에러 판정 + 코드 수정
형식: `- [ ] {탐지ID} {N}건 {오탐/정탐-수정/정탐-한계} → {수정 내용}`

### IL 기록 (pptx-inspection-log.md)
형식: `- [ ] IL-{NN} {이슈 요약} — slide-{NN} 수정 기록`

### 규칙 갱신 (html-prevention-rules.md / nanoBanana-guide.md)
형식: `- [ ] {규칙 파일}: {추가/수정한 규칙 내용}`

### 코드 수정 검증 계획
수정된 코드의 동작을 프로덕션 완료 후 검증. 미검증 `[ ]` 항목은 다음 프로덕션 완료 시 재확인.
형식:
```
### V-{NN}: {검증 제목}
**수정 내용**: {어떤 코드를 왜 수정했는지}
**검증 방법**: {구체적 실행 명령 + 확인 사항}
**통과 기준**: {정량적 기준}
- [ ] 검증 완료
```

| 이벤트 | 체크박스 섹션 | 형식 |
|--------|-------------|------|
| PF/VP/COM/IV/IP/IC/VQA ERROR/WARN/FAIL 발견 | 3분류 판정 + 해당 체크리스트 | `{탐지ID} {N}건 {오탐/정탐-수정/정탐-한계} → ...` |
| pptx-inspection-log.md 기록 | IL 기록 | `IL-{N} {이슈 요약}` |
| 규칙 파일 추가/수정 | 규칙 갱신 | `{파일명}: {규칙 내용}` |
| 탐지 코드 또는 생성 코드 수정 | 코드 수정 검증 계획 | `V-{NN}: {검증 제목}` |
| 이미지 재생성 필요 | 이미지 검수 기록 | `slide-{NN}: {사유}` |

| 사용자 피드백으로 슬라이드 수정 | 로그 기록 상태 | `{수정 내용 요약}` |

### 이미지 검수 기록 (IP/IV/VQA/IC)

Step 1.5B 이미지 생성 + 품질 게이트 결과를 progress.md에 기록:
```markdown
## 이미지 검수 기록 (IP/IV/VQA/IC)
- [x] slide-01 cover: IP PASS → IV PASS → VQA 25.5/27.5 PASS → IC PASS (Tier 1, 금융가 스카이라인)
- [x] slide-03 section: IP PASS → IV PASS → VQA 24.0/27.5 PASS → IC PASS (Tier 1, 공장 내부)
- [ ] slide-08 bg: IP PASS → IV FAIL(IV-02 dark) → 재생성 중
- [x] slide-08 bg: IP PASS → IV PASS(2차) → VQA 23.5/27.5 PASS → IC PASS (muted 톤 강화)
```

**VQA 점수 기록 필수**: 각 이미지의 VQA total을 기록하여 세션 간 품질 추적 가능.
- VQA 23+ → `PASS`
- VQA 20-22 → `WARN` (사용 가능하나 재생성 검토)
- VQA < gate → `FAIL` (재생성 필수)
- VQA < 15 → `DERAIL` (절대 FAIL, 즉시 재생성)

**IC 검증 시점**: Step 2.5/6-3 COM 비교 시 IC-01~IC-04 체크리스트 실행 (`pf-step-2-2.5.md` §IC 검증 체크리스트 참조).
IC 결과도 progress.md에 기록. IC FAIL 시 이미지 재생성 또는 HTML 보정 후 COM 재확인.

**기록 시점**: Step 완료 직후, 다음 Step 진행 전에 반드시 기록. 특히 아래 게이트 단계는 "통과" 기록이 없으면 다음 단계 진행 금지:
- Step 2.5 COM 비교 검증
- Step 6-3 COM 비교 검증

## 세션 복원 시 체크포인트 검증 (필수)

기본 체크포인트 테이블 → `rules/presentation-flow.md` §세션 복원.
아래는 추가 주의사항:

1. **최우선**: `slides/프레젠테이션명/progress.md`를 Read로 읽는다 (대화 요약보다 우선)
2. progress.md가 없으면 대화 요약 + 파일 시스템 상태로 판단
3. **"진행" 응답이 아웃라인 승인인지 초안 승인인지 구분 필수.** 아웃라인 승인 = Step 1.5A로 진행. 초안 승인 = Step 2로 진행.
4. **변환 성공 ≠ COM 비교 통과**: 대화 요약에 "변환 성공"이라고 되어 있어도 progress.md에 "COM 비교 통과" 기록이 없으면 Step 2.5/6-3 재실행

## 통합 검증 파이프라인 (이미지 + HTML + PPTX)

```
아웃라인 (Tier/유형 판정, Visual 메타 태그)
  ↓
IP (프롬프트 검증) ←→ PF (HTML 검증)과 동일 위치: 생성 전 게이트
  ↓
이미지 생성 (Gemini API)
  ↓
IV (자동 메타데이터 검증) ←→ VP (XML 검증)과 동일 위치: 생성 후 자동 체크
  ↓
이미지 품질 게이트 (Read 시각 검사)
  ↓
HTML 슬라이드에 이미지 삽입
  ↓
PF (HTML 전체 검증, 이미지 포함)
  ↓
PPTX 변환
  ↓
VP + VC + IC (PPTX 검증, 이미지 렌더링 포함) ←→ IC는 기존 VC에 통합
  ↓
HTML↔PPTX Visual 비교 (최종 확인)
```

## 흐름 원칙

- **자동 진행**: 각 단계 완료 후 다음 단계 안내를 자연스럽게 이어간다
- **선택지 제공**: 막히는 지점에서는 번호 선택형으로 안내
- **파일 경로 명시**: 항상 전체 경로를 알려주어 바로 더블클릭 가능하게
- **전문 용어 최소화**: "빌드", "변환" 등은 풀어서 설명
- **검사 기록 순환**: 생성 → 검사 → 기록 → 다음 생성 시 참조 (피드백 루프)
- **모든 PPTX 수정은 기록 필수**: 워크플로우 Step 외 즉석 수정도 반드시 `pptx-inspection-log.md`에 기록
- **완료 후 다음 제안**: "다른 형식도 필요하시면" / "다른 주제 프레젠테이션도 만들 수 있어요"
