# 프레젠테이션 제작 보고 규칙

자동승인 모드에서도 보고는 생략하지 않습니다.

## 약어 사전

| 약어 | 풀네임 | 설명 |
|------|--------|------|
| PF | Preflight | HTML 정적 검증 (폰트, overflow, 대비 등) |
| VP | Validate PPTX | PPTX XML 구조 검증 |
| COM | Compare | HTML↔PPTX 시각 비교 |
| IC | Image in Context | 이미지-슬라이드 맥락 일치 검증 |
| IV | Image Validation | 이미지 품질 검증 (크기, 밝기, 포맷) |
| IP | Image Prompt | 이미지 생성 프롬프트 검증 |
| VQA | Visual QA | AI 기반 이미지 품질 점수 (27.5점 만점) |
| IL | Inspection Log | PPTX 변환 이슈 패턴 기록 |
| V-NN | Verification | 탐지 코드 수정 후 검증 항목 |
| C-NN | Change-log | 코드 변동 기록 항목 |

## 보고 원칙

- 각 단계의 시작과 끝에 상태를 알림
- 오류 시 즉시 보고 + 자동 수정 결과 포함
- "정상" 대신 구체적 숫자 사용
- progress.md 갱신과 사용자 보고 모두 수행

## Step별 보고 항목

### Step 0 소스 확인
완료 후: 소스 방식, 수집 결과(건수/글자수/시간), NLM URL(해당 시)

### Step 1 아웃라인
완료 후: 슬라이드 수/섹션, 이미지 계획(Tier별), 비율 분포, 디자인 모드

### Step 1.5A 초안
완료 후: 생성 장수/시간, 다운로드 링크

### Step 1.5B 이미지 생성
시작: "이미지 N장 Gemini 생성 (체인: 켜짐/꺼짐)"
완료 후 필수 항목:

| 항목 | 예시 |
|------|------|
| 생성 결과 | "8장 중 7장 성공, 1장 실패 (slide-12: 안전 필터)" |
| IP/IV | "IP 통과, IV 7장 통과 1장 실패→재생성" |
| VQA | "avg 26.1/30, min 22.5 (slide-8), 전부 통과" |
| 키워드 | "DB 12개→중복제거 8개, avg 2.3개/장" |
| 3분류 판정 | "IV-02 1건 정탐-수정→생성코드 수정" 또는 "판정 없음" |
| 코드 수정 | "생성코드 1건 (nanoBanana-guide.md)" 또는 "없음" |
| 시간 | "2분 15초" |

이미지별 상세 테이블 필수: `| 슬라이드 | 비율 | 해상도 | IP | IV | 품질 |`
5장 이상 시 3장마다 진행 보고.

### Step 2 HTML 생성
완료 후: 생성 결과, PF ERROR/WARN 건수, 3분류 판정 (있으면), 코드 수정, 폰트 검사, 시간
15장 이상 시 5장마다 진행 보고.

### Step 2.5 자동 검증
시작: "검증 시작 (HTML검사→시험변환→PPTX검사→스크린샷 비교)"
완료 후: PF ERROR/WARN 건수, 변환 결과, VP ERROR/WARN 건수, 대비, IC, COM 결과
WARN/ERROR 발생 시: 3분류 판정 + 수정 대상 보고 (아래 §이슈 판정 보고)
COM 차이 발견 시: `| 슬라이드 | 차이 | 수정 |` 테이블
게이트 통과: "검증 통과 — progress.md 기록" 필수

### Step 3 에디터
완료 후: 로컬 URL, 외부 터널 URL, 브라우저 오픈 여부

### Step 4 수정 (사용자 피드백 포함)
수정 후: 수정 슬라이드 목록, 수정 내용, 레이아웃 수정 시 비교 결과

**사용자 피드백 수신 시 필수 보고** (수정 전에 먼저 보고):
```
사용자 피드백 판정:
- 이슈: {설명}
- 판정: {정탐-수정 / 정탐-한계} (사용자 피드백 = 오탐 불가)
- 파이프라인 귀속: {어떤 파이프라인이 잡았어야 하는가} → {PF/VP/COM/없음}
- 놓친 이유: {임계값 미달 / 규칙 미존재 / 엔진 한계 등}
- 수정 대상: {HTML / 탐지코드 / 생성 규칙 / 없음(한계)}
```

### Step 5 출력 선택
"어떤 형식으로 만들까요? (1. PPTX / 2. PDF / 3. 둘 다)"

### Step 6 PPTX 생성
변환: 성공 장수/시간, 파일 크기
검사: PF ERROR/WARN 건수, VP ERROR/WARN 건수, 대비
WARN/ERROR 발생 시: 3분류 판정 + 수정 대상 보고
게이트 통과 필수 → 파일 경로 + 다운로드 링크

### Step 7 PDF 생성
완료 후: 페이지 수/크기/시간, 다운로드 링크

### Step 7.5 프로덕션 후 검증
V-NN 항목별 실행 결과 보고:

| 유형 | 보고 내용 |
|------|----------|
| 회귀 테스트 | "PF 회귀: {N} passed, {N} failed, {N} known issues" |
| change-log 검증 | "C-01~C-{NN}: 실행 가능 {N}건 통과, 행동 검증 {N}건 이월, change-log.md 삭제" |
| 스트레스 테스트 | "PF-{NN}: tests/stress-slides/{dir}/ — TP {N}건 WARN, FP {N}건 미탐지" |
| 생성 규칙 검증 | "이월: 다음 프로덕션에서 확인" (유일한 이월 허용 유형) |

전부 완료 후: "V-NN 전부 통과 — 프로덕션 후 검증 완료"

## 이슈 판정 보고 (3분류 — 모든 파이프라인 공통)

PF/VP/COM/IV/IP/VQA에서 ERROR/WARN 발견 시, 또는 사용자 피드백 수신 시:

```
이슈 판정 보고:
- 발견: {탐지ID} {N}건 (슬라이드 {목록})
- 판정: {오탐 / 정탐-수정 / 정탐-한계}
- 근거: {왜 이 판정인지}
- 파이프라인 귀속: {어떤 파이프라인이 잡았어야 하는가}
- 체크리스트: progress.md 이슈 #{N} 생성 완료
```

판정별 후속 보고:

### 오탐 / 정탐-수정 → 코드 수정 보고 (A~I 완료 시)

```
파이프라인 수정 보고:
- 판정: {오탐 / 정탐-수정}
- 수정 대상: {탐지코드(오탐) / 생성코드(정탐-수정)} ({파일명})
- 수정 내용: {변경 전} → {변경 후}
- 반영: ① 코드 ② IL기록(IL-{NN}) ③ 규칙파일 ④ change-log(C-{NN})
- 검증: 회귀 {N} passed, 스트레스 TP {N}/FP {N}
- V-NN: progress.md V-{NN} 등록
- 승격: {후단계→전단계 승격 여부}
```

### 정탐-한계 → 간소화 보고 (A~D 완료 시)

```
정탐-한계 보고:
- 판정: 정탐-한계 — {한계 원인 (엔진/모델)}
- IL 기록: IL-{NN} ("정탐-한계" 명시)
- 회피 규칙: {추가한 규칙 또는 "기존 규칙으로 충분"}
- 심각도: {ERROR→WARN 강등 여부} (ERROR인 경우만)
```

## 최종 완료 보고

```
프레젠테이션 완료:
- 제목: {명}, 슬라이드 {N}장, 이미지 {N}장
- 출력: PPTX ({크기}) + PDF ({크기})
- 검증: PF ERROR {N}, VP ERROR {N}
- 이슈: 오탐 {N}건, 정탐-수정 {N}건, 정탐-한계 {N}건
- 코드 수정: C-01~C-{NN} ({N}건, 검증 후 삭제)
- V-NN: 회귀 {passed}/{failed}, 스트레스 {N}건 통과
- 시간: ~{N}분
```
