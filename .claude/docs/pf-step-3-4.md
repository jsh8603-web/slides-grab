# Step 3~4 — 비주얼 에디터 + 수정 반복

## Step 3 — 비주얼 에디터 실행

슬라이드 생성 완료 후 비주얼 에디터를 실행한다 (viewer.html 미리보기 대신).

### 기존 에디터 종료 (필수)

새 프레젠테이션 에디터를 시작하기 전에, 기존에 실행 중인 에디터를 먼저 종료한다.
다른 프레젠테이션의 에디터가 남아있으면 혼동을 유발하므로, 반드시 기존 에디터를 닫고 새 에디터를 실행한다.

```bash
# 기존 에디터 종료 — 3456~3460 범위의 모든 에디터 프로세스 종료
netstat -ano | grep "LISTENING" | grep -E ":(3456|3457|3458|3459|3460)" | awk '{print $5}' | while read pid; do taskkill //PID $pid //F 2>/dev/null; done
```

Step 1.5A(초안 파일 서빙), Step 3(에디터 실행) 모두 서버 시작 전 동일하게 적용한다.

### 포트 충돌 처리

기본 포트(3456)에 이미 다른 프레젠테이션 에디터가 실행 중일 수 있다.
실행 전 포트 사용 여부를 확인하고, 충돌 시 다음 포트(3457, 3458, ...)로 자동 전환한다.

```bash
# 1. 기본 포트 확인
curl -s http://localhost:3456/ > /dev/null 2>&1

# 2-a. 포트 비어있으면 기본 포트 사용 (GEMINI_API_KEY 필수 전달, --tunnel 추가)
GEMINI_API_KEY=$GEMINI_API_KEY node scripts/editor-server.js --slides-dir slides/프레젠테이션명 --port 3456 --tunnel &

# 2-b. 포트 사용 중이면 다음 포트로 실행
GEMINI_API_KEY=$GEMINI_API_KEY node scripts/editor-server.js --slides-dir slides/프레젠테이션명 --port 3457 --tunnel &

# 3. 서버 시작 대기 후 브라우저 자동 오픈 (필수)
sleep 2
start http://localhost:{포트}/ 2>/dev/null || powershell -Command "Start-Process 'http://localhost:{포트}/'" 2>/dev/null
```

**`--tunnel` 필수 추가**: cloudflared 터널이 자동으로 열리고 콘솔에 외부 접속 URL이 출력된다.

### 에디터 실행 절차 (필수)

1. `node scripts/editor-server.js`를 백그라운드(`&`)로 실행
2. `sleep 2`로 서버 시작 대기
3. **브라우저 자동 오픈 필수**: `start` 또는 `powershell Start-Process`로 브라우저에서 에디터 URL을 연다
4. 사용자에게 안내 메시지 출력

**브라우저 오픈 없이 URL만 안내하는 것은 금지.** 사용자가 직접 URL을 복사할 필요 없도록 자동으로 열어준다.

### 안내 메시지

실행 후 사용자에게 실제 할당된 포트 번호 + 터널 URL로 안내:

```
슬라이드가 완성됐습니다! 비주얼 에디터가 브라우저에서 열렸습니다.

로컬: http://localhost:{포트}/
외부: {터널URL} (모바일/외부 접속용)

에디터에서 슬라이드를 클릭하면 직접 텍스트를 수정할 수 있고,
영역을 드래그하면 AI에게 해당 부분 수정을 요청할 수 있습니다.

수정이 끝나면 알려주세요!
```

**터널 URL 확인**: 에디터 서버 시작 로그에서 `Tunnel:` 줄의 URL을 읽어 안내한다.

## Step 4 — 수정 반복

사용자 피드백에 따라 슬라이드 수정. 에디터가 실행 중이면 브라우저에서 바로 반영됨 (fsWatch → SSE 자동 갱신).

### 사용자 피드백 → 3분류 판정 (필수 — 수정 전에 반드시 실행)

**사용자 피드백 = 무조건 정탐** (오탐 불가). 슬라이드 HTML 수정 전에 반드시:
1. **3분류 판정**: 정탐-수정 / 정탐-한계 판정
2. **파이프라인 귀속**: "어떤 파이프라인이 이걸 잡았어야 하는가?" 판단
3. **체크리스트 생성**: progress.md에 해당 체크리스트(A~I 또는 간소화 A~C) 추가
4. **A. 판정 완료**: `[x] A. 판정:` 체크 후에만 HTML 수정 시작

**하드 가드**: `checklist-guard.mjs`가 슬라이드 HTML 수정을 감시. progress.md에 판정(A항목) 미완료 이슈가 있으면 Edit/Write 차단.

상세: `CLAUDE.md` §공통 절차 참조.

### 수정 피드백 형식 (필수)

사용자 수정 요청 시, AI가 수정 전에 변경 내용을 번호 리스트로 정리하여 안내한 뒤 진행 여부를 확인한다:

```
수정 내용:
1. 슬라이드 3: 제목 폰트 크기 24pt → 20pt
2. 슬라이드 7: 차트 데이터 업데이트 (2025 → 2026)
3. 슬라이드 12: 이미지 교체

진행할까요?
```

사용자가 직접 대화형으로 수정 지시 가능. AI가 HTML 수정 → 에디터 자동 갱신 → 사용자 브라우저 새로고침으로 즉시 확인.

### 수정 유형별 재검증 매트릭스

모든 수정은 아래 매트릭스에 따라 재검증한다. **Step 5 진행 전 재검증 완료 필수.**

| 수정 유형 | PF | VP | COM | IV/IC | VQA | 비고 |
|----------|:--:|:--:|:---:|:-----:|:---:|------|
| 텍스트 내용 변경 (오타, 수치) | — | — | — | — | — | 에디터 확인만 |
| 폰트/여백/레이아웃 변경 | O | O | O | — | — | convert-native 재실행 |
| 색상/배경 변경 | O | O | O | — | — | VP-04 대비 재확인 |
| 이미지 교체 (NanoBanana 재생성) | O | — | O | O | O | 아래 §이미지 교체 프로세스 |
| 이미지 교체 (사용자 제공) | O | — | O | — | — | 대비/비율만 확인 |
| 슬라이드 추가/삭제 | O | O | O | — | — | 전체 재변환 |
| CSS 구조 변경 (grid, flex) | O | O | O | — | — | 변환 충실도 위험 높음 |

O = 재실행 필수, — = 불필요

### 수정 후 재검증 절차 (체크리스트)

레이아웃/구조/이미지 수정 시 아래 체크리스트를 순서대로 실행 (**완료 게이트**: 전부 `[x]` 전까지 다음 작업 차단):

```
- [ ] 1. HTML 수정 적용
- [ ] 2. PF 재실행: `node scripts/preflight-html.js --slides-dir ...`
      → ERROR 0건 확인 (ERROR 시 HTML 재수정)
- [ ] 3. PPTX 재변환: `node scripts/convert-native.mjs --slides-dir ... --output ...`
      → FAILED 0건, VP ERROR 0건 확인
- [ ] 4. COM 프리뷰 생성 (수정된 슬라이드만):
      powershell -ExecutionPolicy Bypass -File scripts/export-slides-png.ps1 \
        -PptxPath "PPTX경로" -OutputDir "preview경로" -Slides "수정번호"
- [ ] 5. HTML↔PPTX 비교: Read로 열어 형태변형/텍스트잘림/색상차이 없음 확인
- [ ] 6. 이슈 발견 시: 수정→재변환→재확인 (최대 2회)
- [ ] 7. 수정 내용을 pptx-inspection-log.md에 기록 (레이아웃 오류 관련 시)
- [ ] 8. progress.md에 재검증 완료 기록
```

### 이미지 교체 프로세스

사용자가 이미지 교체를 요청하면 아래 분기를 따른다:

#### A. NanoBanana 재생성 (AI 이미지)

```
- [ ] 1. 사용자 피드백 반영하여 프롬프트 수정 (slide-outline.md NanoBanana 태그)
- [ ] 2. 재생성 실행:
      node scripts/generate-images.mjs \
        --outline slide-outline.md \
        --output slides/프레젠테이션명/assets \
        --regenerate {슬라이드번호}
- [ ] 3. IP/IV 자동 검수 통과 확인 (콘솔 출력)
- [ ] 4. Read로 이미지 시각 검사 (기준 1-5: 가짜데이터, 한글, 허용유형, 맥락, 규칙)
- [ ] 5. VQA 스코어 확인 (score >= 20/27.5)
- [ ] 6. HTML 슬라이드에 이미지 경로 반영 (필요 시)
- [ ] 7. 이미지-텍스트 대비 확인 (text-shadow/오버레이 필요 시 추가)
- [ ] 8. 재검증 절차 실행 (위 §수정 후 재검증 절차 체크리스트)
```
**완료 게이트**: 전부 `[x]` 전까지 다음 작업 차단.

#### B. 사용자 제공 이미지

**완료 게이트**: 전부 `[x]` 전까지 다음 작업 차단.
```
- [ ] 1. 이미지를 assets/ 폴더에 저장
- [ ] 2. HTML 슬라이드에 이미지 경로 반영
- [ ] 3. 이미지-텍스트 대비 확인 (밝은 이미지 옆 텍스트 → text-shadow/오버레이)
- [ ] 4. 이미지-컨테이너 비율 확인 (비율 불일치 시 이미지 리사이즈 또는 컨테이너 조정)
- [ ] 5. 재검증 절차 실행 (위 §수정 후 재검증 절차 체크리스트)
```

#### C. 이미지 삭제 (NanoBanana → HTML 다이어그램 전환)

**완료 게이트**: 전부 `[x]` 전까지 다음 작업 차단.
```
- [ ] 1. 이미지 파일 삭제 + HTML에서 <img> 제거
- [ ] 2. HTML/CSS로 대체 시각 요소 구현
- [ ] 3. slide-outline.md NanoBanana 태그를 "없음"으로 변경
- [ ] 4. 재검증 절차 실행
```

### 오류 관련 수정 시 — 검사 기록 필수

사용자 요청으로 수정한 내용이라도 **레이아웃 오류와 관련된 수정이면** `.claude/docs/pptx-inspection-log.md`에 기록한다.
기록 대상: 오버플로, 텍스트 잘림, 요소 겹침, 하단 넘침, 가독성 저하 등 렌더링 품질 이슈.
기록 제외: 텍스트 내용 변경, 색상 취향 변경, 슬라이드 순서 변경 등 디자인 선호 이슈.

### 탐지 결과 승격 (Step 4에서도 적용)

수정→재검증 과정에서 반복 발견된 COM/VP 패턴은 PF/VP 규칙으로 승격 검토한다.
상세 기준 → `CLAUDE.md` §탐지 결과 승격.

"없어" / "완료" / "괜찮아" 등 수정 없음 의사 표현 시:
1. **에디터 서버는 유지** — Step 6/7 산출물 생성 + 외부 다운로드를 위해 종료하지 않는다.
2. **Step 5 선택형 질문**을 한다.

**절대 Step 5를 건너뛰고 변환을 시작하지 않는다.** 사용자가 명시적으로 형식을 선택할 때까지 대기.
